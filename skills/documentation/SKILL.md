# Documentation

Write for the reader who does not have the context you have.

1. **Identify the audience** (end user, contributor, maintainer) and what they need to accomplish.
2. **Lead with purpose** — state what the thing is and what problem it solves in the first sentence.
3. **Show, then tell.** Provide a concrete, runnable example early.
4. **Structure** with clear headings so readers can scan: overview, install/setup, usage, examples, configuration, troubleshooting.
5. **Document the interface**: inputs, outputs, errors, and side effects — not the implementation.
6. **Keep it current.** Update docs in the same change as the code they describe.

## README essentials

- One-line description of the project.
- Quick start: install and run in the fewest steps.
- Usage examples with expected output.
- Configuration and requirements.
- How to run tests / contribute.

## Guidelines

- Prefer short sentences and active voice.
- Every code example should actually work.
- Comment the **why**, not the obvious **what**. Do not add noise comments.
- Avoid marketing language; be precise and concrete.
